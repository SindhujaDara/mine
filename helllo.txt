What is Git?
Git is a distributed version control system used to track changes in source code.
It lets you:
Save versions of your project
Go back to previous versions
Work on branches
Collaborate without overwriting each other's work
Git works on your local computer.

GitHub?
GitHub is an online platform that hosts Git repositories.
It lets you:
Store your Git repositories in the cloud
Collaborate with others
Review code
Manage projects (issues, pull requests, etc.)
GitHub is a website/service that works using Git.

Git	GitHub differences
Type	Version control tool	Hosting & collaboration platform
Where it works	Local computer	Online (cloud)
Purpose	Track project history	Store/share Git repositories
Works without the Internet?	Yes	No
Created by	Linus Torvalds	Microsoft (owns GitHub)

What is Source Code Management (SCM) on GitHub?
Source Code Management (SCM) on GitHub means using GitHub to:
Store your source code
Track all changes
Collaborate with others
Review and merge updates
Manage versions of your project
GitHub acts as an online storage + collaboration system for your Git repositories.

How Source Code Management Works on GitHub
1. Create a repository (repo)
A repo is like a folder that stores your code and history.
2. Push your code to GitHub
Upload your project using Git.
3. track changes
Every update you make is stored as a commit.
4. Work with branches
Branches allow separate development without messing up the main code.
5. Create Pull Requests (PR)
Ask to merge branch changes into the main project.
6. Merge changes
Approved PRs are merged into the main branch.
7. Collaborate with team
Teams can comment, review, and contribute.

# **What is a Docker Container Image?**

**Simple meaning:**
A **Docker image** is a *blueprint / template / model* used to create containers.

Think of it like:

* **Image = Recipe**
* **Container = Dish prepared from that recipe**

### Example

You write in your Dockerfile:

```
FROM python
COPY app.py .
CMD ["python", "app.py"]
```

This **Dockerfile** ‚Üí builds an **Image** ‚Üí from that image you run **containers**.

---

#  **Why do we need a Docker Image?**

Because:

1Ô∏è It contains **your app + dependencies**
(Example: Python version, libraries, app files)

2Ô∏è It ensures the app runs **exactly the same on any system**
Laptop, server, cloud ‚Üí same result.

3Ô∏è It allows you to **quickly create containers**
No need to re-setup every time.

---

#  **What is a Docker Container? (Simple)**

Container = **Running instance** of the Image.
Image is **static**, Container is **active**.

Example:

* Image = a cake mold
* Container = actual cake made from it.

---

#  **Difference: Image vs Container (Super Simple)**

| Image                 | Container           |
| --------------------- | ------------------- |
| Blueprint             | Running application |
| Stored file           | Live instance       |
| Cannot run by itself  | Runs app            |
| Built from Dockerfile | Created from image  |

---

# üõ†Ô∏è **Why do we use Docker Tools?**

Docker Tools = Docker Engine, Docker CLI, Docker Desktop, Docker Hub.

We use these because they help to:

### ‚úî Build images

`docker build -t myapp .`

### ‚úî Run containers

`docker run myapp`

### ‚úî Share images (via Docker Hub)

`docker push myapp`

### ‚úî Pull existing images

`docker pull python`

### ‚úî Manage containers & networks

Start, stop, delete, monitor, connect.

---

#  Super Simple Summary

* **Docker Image** ‚Üí Template/Model for your app
* **Docker Container** ‚Üí Running version of image
* **Docker Tools** ‚Üí Help you build, run, share, manage

Docker is a containerization platform that allows you to package an application along with all its dependencies into a lightweight, portable unit called a container.
These containers run reliably and consistently across different computing environments.
Key Terms in Definition

Containerization ‚Üí packaging apps with dependencies

Image ‚Üí template used to create containers

Container ‚Üí running instance of an image

Isolated environment ‚Üí app runs independently of the host OS

Portable ‚Üí same behavior everywhere (Windows, Linux, cloud)
